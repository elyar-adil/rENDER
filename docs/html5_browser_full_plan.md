# rENDER HTML5 浏览器功能完善总计划（严格执行版）

> 目标：将 `rENDER` 从“可渲染部分网页的教学型引擎”推进到“具备主流 HTML5 页面可用性、可测试、可持续迭代”的浏览器内核。  
> 周期：约 36 周（9 个月），按阶段推进，严格以测试与验收门槛为准，不达标不进入下一阶段。

---

## 0. 执行原则（必须遵守）

1. **测试先行**：每个能力点先补测试再实现，不允许“先改功能后补测试”。
2. **增量发布**：每周都有可验证交付（测试通过、截图对比、性能数据）。
3. **无特判策略**：禁止针对某个网站写临时分支逻辑，统一沉淀为通用能力。
4. **回归红线**：新增能力不得造成既有兼容性下降；一旦下降，先修复再继续。
5. **可观测性优先**：所有关键路径必须有日志、指标、错误分类和调试开关。
6. **文档同步**：每次完成里程碑必须更新 `docs/` 下对应设计与测试文档。

---

## 1. 目标定义与验收标准

### 1.1 最终能力目标（HTML5 可用级）

- **HTML**：支持常见文档结构、语义标签、表单控件、媒体标签基础行为。
- **CSS**：支持现代布局（Flex/Grid 已有基础继续完善）、选择器、伪类、层叠与继承细节。
- **JavaScript/DOM**：可运行主流页面初始化脚本、事件驱动交互、定时器、异步网络请求。
- **网络**：稳健的资源加载管线（重定向、缓存、压缩、字符集、并发与超时控制）。
- **渲染**：稳定布局与绘制，具备基础重排重绘机制。
- **平台能力**：错误处理、调试、性能与兼容性回归体系完善。

### 1.2 “完成”判定（必须全部达成）

1. **核心回归测试**：
   - 单元/集成测试通过率 100%。
   - 新增 HTML5 相关测试覆盖目标能力清单。
2. **兼容性基线**：
   - 选定 30 个真实页面样例，首屏结构和主要交互通过定义的断言门槛。
3. **性能基线**：
   - 在同等测试机上，页面首次可绘制时间与重排次数达到阶段 KPI。
4. **稳定性基线**：
   - 连续批量回归（至少 200 页面）无崩溃，错误可归类可复现。

## 2. 36 周路线图（按周执行）

## 阶段 A（第 1-4 周）：基线清点与测试地基

### 周 1：能力盘点与缺口矩阵

- 输出：`docs/html5_gap_matrix.md`
- 内容：按 HTML/CSS/JS/Network/Rendering 分层列出“已支持/部分支持/未支持”。
- 动作：建立优先级（P0/P1/P2），定义每项验收用例。

### 周 2：测试框架升级

- 建立分层测试目录：语法级、DOM API 级、布局级、页面级。
- 增加失败快照（DOM dump + layout tree + screenshot diff）。
- 产出统一测试命名规范与 fixture 模板。

### 周 3：页面回归样本池

- 建立 30 个分级样本页面（新闻、电商、文档、表单、门户）。
- 为每页定义“结构断言 + 交互断言 + 可视断言”。

### 周 4：CI 基线

- CI 强制执行：单测、回归、静态检查、关键截图比对。
- 设定“回归阻断规则”：核心指标下降即禁止合并。

**阶段 A 里程碑**：之后每一行代码都处于可量化回归保护下。

---

## 阶段 B（第 5-10 周）：HTML 与 DOM 语义补齐

### 周 5-6：HTML 解析与树构建规范化

- 补齐常见容错规则（标签自动闭合、错误嵌套修复）。
- 增强文本节点合并、空白处理、实体解析一致性。

### 周 7：表单与交互元素基础语义

- `input/select/textarea/button/form/label` 行为补齐。
- 表单提交、默认值、禁用态、基础校验行为落地。

### 周 8：DOM Core API 完整度提升

- 补齐常用 DOM 查询、节点增删改、属性与 classList 操作。
- 标准化 `innerHTML/textContent` 行为边界。

### 周 9：事件系统 v1

- 捕获/冒泡链路、`stopPropagation`、`preventDefault`。
- 鼠标与键盘基础事件行为统一。

### 周 10：Mutation 与重排触发点梳理

- DOM 变更触发样式失效、布局失效、绘制失效的规则落地。
- 建立失效诊断日志（为何触发 reflow/repaint）。

**阶段 B 里程碑**：页面结构与基础交互脚本可稳定运行。

---

## 阶段 C（第 11-18 周）：CSS 能力强化

### 周 11-12：选择器与优先级

- 补齐属性选择器、组合器细节、伪类常见集合。
- 提升 specificity 与 cascade 冲突处理准确性。

### 周 13：盒模型与尺寸计算

- 完整梳理 `min/max`、百分比尺寸、替换元素尺寸策略。
- 统一 `box-sizing`、边距折叠等边界情况。

### 周 14-15：Flex 深化

- `flex-basis`、伸缩因子、换行、多行对齐、最小内容约束。
- 与绝对定位、overflow、百分比高度协同行为测试。

### 周 16-17：Grid 深化

- 轨道尺寸算法、自动放置、命名网格线与隐式网格。
- `minmax/auto-fit/auto-fill` 常见场景验证。

### 周 18：层叠上下文与绘制顺序

- `z-index`、透明度、定位上下文、裁剪与溢出绘制规则。

**阶段 C 里程碑**：主流现代布局页面首屏结构基本正确。

---

## 阶段 D（第 19-26 周）：JavaScript 运行时与 Web API 实用化

### 周 19-20：JS 语言子集补全（优先实用语法）

- 补齐页面常用语法与对象行为（以真实脚本覆盖驱动）。
- 引入脚本执行错误分类与可读栈信息。

### 周 21：事件循环与任务队列

- 宏任务（`setTimeout`）与微任务（Promise 基础）执行顺序。
- 渲染时机与脚本时机协同策略。

### 周 22-23：DOM 事件与脚本联动

- `addEventListener` 选项、一次性监听、移除监听边界。
- 交互触发 DOM 更新后的渲染一致性。

### 周 24：网络 API 基础

- `XMLHttpRequest/fetch`（可先以核心子集实现）。
- 超时、错误回调、状态码处理。

### 周 25：存储与 URL 能力基础

- `localStorage/sessionStorage` 基础行为。
- URL 解析、查询参数处理一致性。

### 周 26：脚本安全与隔离边界

- 同源策略基础约束、资源访问限制。
- 危险接口显式禁用并给出可诊断错误。

**阶段 D 里程碑**：大部分“页面能打开但交互失效”的问题被消除。

---

## 阶段 E（第 27-32 周）：网络、资源与性能体系

### 周 27：资源加载调度

- CSS/JS/Image 并发、优先级和依赖关系。
- 防止阻塞链过长导致白屏。

### 周 28：缓存与复用

- 内存缓存 + 条件请求（ETag/Last-Modified）基础。
- 重复访问性能验证。

### 周 29：编码与压缩

- 字符集识别与解码回退。
- gzip/br 等压缩处理。

### 周 30：容错与恢复

- 网络重试策略、超时策略、降级策略。
- 局部资源失败不导致整页崩溃。

### 周 31-32：性能专项

- 引入性能剖析点：解析、样式、布局、绘制、脚本执行。
- 目标：关键页面 TTFR（首次可见渲染）降低 30%（相对阶段 A 基线）。

**阶段 E 里程碑**：打开页面速度与稳定性达到“可日常试用”。

---

## 阶段 F（第 33-36 周）：收敛、验收与发布

### 周 33：兼容性冲刺

- 对 30 个样本页逐项清缺口，P0 全清，P1 清理到可接受阈值。

### 周 34：稳定性压测

- 批量打开 200+ 页面进行长稳测试。
- 收敛内存泄漏、崩溃、死循环问题。

### 周 35：文档与开发者体验

- 完成架构图、模块 API 文档、调试指南、问题定位手册。

### 周 36：冻结与发布

- 版本冻结，回归全跑，形成发布说明与后续路线图（P2+）。

**阶段 F 里程碑**：形成“功能完善的 HTML5 可用浏览器内核版本”。

## 3. 工作分解结构（WBS）

每个能力项都必须经过以下流水线：

1. **需求定义**：明确规范范围与非目标。
2. **测试设计**：最小用例 + 真实页面回归 + 反例。
3. **实现**：小步提交，单功能单提交。
4. **验证**：自动化 + 手工抽检。
5. **文档**：设计决策、已知限制、后续改进。

建议每周节奏：

- 周一：缺陷与需求分解
- 周二到周四：实现 + 单测
- 周五：回归 + 文档 + 复盘

---

## 4. 指标看板（每周必更新）

- 功能覆盖：HTML/CSS/JS API 支持项数量与完成率。
- 测试健康：总数、通过率、波动趋势、慢测排行。
- 兼容性：30 样本页通过率、阻断缺陷数。
- 性能：TTFR、总渲染耗时、重排次数、脚本耗时。
- 稳定性：崩溃数、超时数、内存增长曲线。

---

## 5. 风险与预案

1. **范围膨胀风险**：
   - 预案：严格 P0/P1/P2；P2 一律不阻塞发布。
2. **JS 复杂度过高**：
   - 预案：先以真实页面脚本覆盖驱动“实用子集”，避免盲目追全量规范。
3. **性能退化**：
   - 预案：每周性能回归，超过阈值立即停新功能修性能。
4. **回归噪声大**：
   - 预案：构建稳定 fixture，拆分不稳定用例并标注原因。
